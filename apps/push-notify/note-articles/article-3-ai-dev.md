# コミットの半分はAIが書いた——5時間でアプリをApp Storeに出した全記録

先日、自分が作ったアプリの git 履歴（コードの変更記録）をぼんやり眺めていたら、あることに気づきました。

コミットメッセージ（変更を保存するときに書く一言メモ）の末尾に、こんな一行がついているものがやたら多いんです。

```
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

数えてみると、全63コミットのうち **32コミット——半分以上がAIとの共著** でした。

これは「ChatGPTにコードを聞いてコピペした」という話ではありません。AIがプロジェクトのファイルを直接読み書きし、git にコミットまでしてくれる。**人間のペアプログラマーと同じように一緒に開発した記録** が、コミット履歴にそのまま残っているんです。

このアプリの名前は「かんたんプッシュ」。API（外部からプログラムを呼び出す仕組み）を1回叩くだけで、スマホにプッシュ通知が届くiOSアプリです。サーバー監視やIoT、GitHub Actionsなど、ちょっとした通知を飛ばしたいエンジニア向けに作りました。

そして、このアプリの **実際のコーディング時間は合計で約5時間** です。

この記事では、AIコーディングアシスタント「Claude Code」と一緒にアプリを開発し、App Storeに公開するまでの全工程を書きます。うまくいったことだけでなく、**失敗した場面も正直に** 記録しました。

## Claude Codeとは何か——ターミナルの中にいるペアプログラマー

Claude Codeを知らない方のために、ざっくり説明します。

ChatGPTやClaudeのチャット画面を使ったことがある方は多いと思います。「こういうコードを書いて」とお願いすると、コードを生成してくれるアレです。

Claude Codeは、それの **ターミナル版** です。

ターミナルというのは、プログラマーがコマンド（命令文）を打ち込む黒い画面のこと。Claude Codeはこのターミナルの中で動いて、プロジェクトのファイルを **直接読み書き** してくれます。

チャット画面のAIだと「コードをコピー→自分のエディタに貼り付け→動かない→チャットに戻ってエラーを伝える→修正コードをまたコピー……」という往復が発生します。Claude Codeにはそれがありません。AIが自分でファイルを開き、コードを編集し、場合によってはビルド（プログラムの構築）やテストまで実行してくれます。

ひとことで言えば、**自分のパソコンの開発環境に同居して一緒にコードを書いてくれるAIペアプログラマー** です。

### 成功例2: SwipeableRow——70行の複雑なジェスチャーUI

次に驚いたのが、UIコンポーネントの品質です。

iPhoneのメールアプリで、メールを左にスワイプすると「ゴミ箱」ボタンが出てきますよね。あの操作を受信箱の通知一覧に実装したかったのですが、React Nativeでこれをゼロから書くのは相当面倒です。

**私が送ったプロンプト:**

```
受信箱の各通知を左にスワイプすると削除ボタンが出るUIを作って。
- iOSメールアプリのような操作感
- ゆっくりスワイプ → 赤い「削除」ボタンが見える
- 素早くスワイプ → 一気に削除ボタン位置まで開く
- PanResponderとAnimated APIで、useNativeDriver有効にして
```

**AIが返してきたコード（抜粋）:**

```typescript
const panResponder = useRef(
  PanResponder.create({
    // 水平方向のスワイプだけ反応する（縦スクロールと競合しない）
    onMoveShouldSetPanResponder: (_, g) =>
      Math.abs(g.dx) > 4 && Math.abs(g.dx) > Math.abs(g.dy) * 1.2,

    // 指の動きに追従してスライド
    onPanResponderMove: (_, g) => {
      const next = offsetRef.current + g.dx;
      translateX.setValue(Math.max(Math.min(next, 0), -SCREEN_W));
    },

    // 指を離したときの判定
    onPanResponderRelease: (_, g) => {
      const next = offsetRef.current + g.dx;
      if (g.vx < -0.5 || next < -BUTTON_W * 0.4) {
        snap(-BUTTON_W);  // 速いスワイプ or 十分な距離 → 削除ボタン表示
      } else {
        snap(0);          // それ以外 → 元に戻る
      }
    },
  })
).current;
```

このコードの何がすごいかというと——

- `g.vx < -0.5`（スワイプ速度が秒速0.5以上なら即反応）で「サッと素早くスワイプ」を検知
- `Math.abs(g.dx) > Math.abs(g.dy) * 1.2` で水平方向の動きだけ拾い、縦スクロールと衝突しない
- `useNativeDriver: true` でJavaScriptスレッドを介さず60fpsのアニメーションを実現

PanResponder（タッチ操作を検知するAPI）のドキュメントを読んで、速度判定や方向判定のロジックを自分で設計していたら、半日はかかっていたと思います。それが **プロンプト4行で、動くコードとして出てきた** わけです。

## AIに任せて失敗したこと——人間の判断が必要だった3つの場面

成功談だけでは信用していただけないと思うので、失敗も正直に書きます。

### 失敗1: プッシュ通知の受信設計を間違えた

最も手痛かった失敗は、通知の受信ハンドラ（受信時に動く処理）の設計です。

AIが最初に提案したのは、「通知タップのリスナー（待ち受け処理）を登録して、タップされたら反応する」という方式でした。一見正しそうですが、**アプリが完全に終了している状態で通知をタップすると、リスナーの登録が間に合わず通知を取りこぼす** ケースがあったんです。

これは実機でテストして初めてわかりました。

結局、`getLastNotificationResponseAsync()` という「最後にタップされた通知をいつでも取得できる」関数を使う方式に **私の判断で** 切り替えました。

```typescript
// 修正後の設計: リスナーに頼らず、能動的に取りに行く
const lastResponse = await getLastNotificationResponse();
if (lastResponse) {
  const responseId = lastResponse.notification?.request?.identifier;
  if (responseId && responseId !== processedResponseIdRef.current) {
    processedResponseIdRef.current = responseId;
    const notif = extractNotification(lastResponse);
    // ...
  }
}
```

さらに、通知センター（iPhone上部から引き出す通知一覧）に残っている未読通知も取り込む「二重取り込み戦略」を追加しました。タップされた通知 + 通知センターの残留通知、この両方を拾うことで取りこぼしをゼロにしています。

> **教訓:** アーキテクチャ（システム全体の設計方針）の選択は、まだ人間がすべき仕事です。AIは「動くコード」は書けますが、「実機での挙動の癖」や「エッジケース（まれに起きる例外的な状況）」まで見通すのは難しい。設計はAIに提案してもらい、最終判断は人間が下す——このパターンが今のところ最適だと感じています。

### 失敗2: App Store審査の実務ノウハウはAIにない

App Storeに提出したところ、`ITSAppUsesNonExemptEncryption`（暗号化に関する申告設定）について指摘を受けました。

AIに聞いても「一般的には `false` に設定することが多いです」という教科書的な回答しか返ってきません。実際に必要だったのは、Expoの `app.json` の特定の場所に以下を追加すること。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "ITSAppUsesNonExemptEncryption": false
      }
    }
  }
}
```

たった3行の設定ですが、**「どのファイルの、どの階層に、何を書くか」** はビルド→提出→リジェクト→修正というサイクルを実際に回さないとわかりません。

AIは「App Storeに自分のアプリを提出したことがない」ので、こういう実務的なノウハウには限界があります。審査対応は、人間の経験値がものを言う領域でした。

### 失敗3: UXの細かい調整は「感覚」の世界

フォントサイズの範囲を11pt〜22ptに設定したのは私の判断です。色パレットの7色を選んだのも私です。

「適切なフォントサイズ範囲を決めて」とAIに聞くこともできます。でも返ってくるのは「iOSのHuman Interface Guidelinesでは本文に17ptが推奨されています」のような一般論です。

**自分のアプリの、自分のユーザーにとって最適な値** を決めるには、実機で触って、画面を見て、「これは小さすぎるな」「この色は通知の優先度表示と被るな」と判断する必要があります。

> **教訓:** デザインの「最後の10%」は、AIではなく人間の感覚に頼る部分が大きい。AIに80%を作ってもらい、残り20%を人間が磨く——そのくらいの分担がちょうど良いと感じました。

## 私が実践したAIとの協業テクニック5選

ここからは、2日間の開発で効果があったテクニックを5つ紹介します。

### テクニック1: ファイル名を指定して具体的に指示する

**悪い例:** 「アプリをもっと使いやすくして」
**良い例:** 「InboxScreen.tsxに、通知のタイトルと本文を横断検索する検索ボックスを追加して」

Claude Codeはプロジェクト内のファイルを自動的に読んでくれるので、ファイル名を指定するだけで文脈を理解してくれます。「どのファイルの何を変えるか」を明確にすると、AIの出力精度が格段に上がりました。

### テクニック2: エラーログを丸ごと貼って「修正して」と言う

Day 1で「アプリを開くたびにAPIキーが新しく生成される」というバグに遭遇しました。

私がやったのは、症状をそのまま伝えただけです。

```
アプリを開くたびにAPIキーが再生成されてしまう。
前回保存したキーが読み込まれず、毎回上書きされている。修正して。
```

AIはすぐに原因を特定しました。AsyncStorage（端末内のデータ保存）の読み込みが完了する **前に** キーの有無をチェックしていたため、「キーがない」と判断して毎回新規生成していたんです。

修正はたった1箇所。「読み込み完了を待ってからチェックする」だけ。所要時間2分。

コミットメッセージにはこう書かれています：

```
Fix API key regenerating on every app open
Wait for AsyncStorage to load before checking if key is null.
Previously the key was overwritten before the stored value loaded.
```

原因の特定から修正、コミットメッセージの記述まで、すべてAIがやってくれました。バグの症状だけ伝えればいい——これは本当に楽です。

### テクニック3: 型定義ファイル（types.ts）を最初に渡す

成功例3でも触れましたが、TypeScriptの型定義を先にAIに読ませることは非常に効果的です。

「かんたんプッシュ」の `types.ts` はたった28行ですが、ここに通知の構造（title、message、priority、category……）やフリーミアムの制限値（`FREE_LIMIT = 10`）が定義されています。

この「設計図」をAIが把握している状態でコードを書かせると、**すべての関数が同じ型に従って書かれる** ため、ファイル間の不整合が起きません。

「小さな型定義ファイルが、AIの出力品質を劇的に上げる」——これは予想外の発見でした。

### テクニック4: プロジェクトルール（CLAUDE.md）を活用する

正直に書くと、今回のプロジェクトでは `CLAUDE.md`（Claude Codeが自動的に読むプロジェクトルールファイル）を **使いませんでした**。

その結果、AIが時々「このプロジェクトでは使わないライブラリ」をインポートしようとしたり、コーディングスタイルが微妙にブレたりすることがありました。

次のプロジェクトでは必ず使うつもりです。書く内容は、たとえばこういったものです：

- 「Expo SDK 52を使用。SDK 51以前のAPIは使わない」
- 「スタイルは StyleSheet.create で定義する。インラインスタイルは避ける」
- 「状態管理はReact標準のuseState/useContextで。Reduxは使わない」

プロジェクト固有のルールを明文化しておけば、AIが毎回同じ間違いをしなくなります。人間の新メンバーにドキュメントを渡すのと同じ発想です。

### テクニック5: 「〜しないで」の指示が想像以上に効く

特に大規模な変更をお願いするときに効果的だったのが、ネガティブプロンプト（否定の指示）です。

```
6機能を追加してください。
ただし:
- 既存のテスト用console.logは削除しないで
- RootNavigator.tsxのタブ順序は変えないで
- 色の定数はtheme.tsに集約して、各ファイルにハードコードしないで
```

AIは指示されたことを全力でやろうとするので、「やってはいけないこと」を明示しないと意図しない変更が紛れ込みます。**「変更すべきでない部分」を先に伝えることで、副作用を大幅に減らせました。**

これは6機能を一括実装した成功例3で特に威力を発揮しました。39ファイルを変更する大型コミットでも、触ってほしくない部分がきちんと保護されていたのは、このテクニックのおかげです。

## まとめ——「作れない」から「作れる」時代へ

この記事で伝えたかったのは、2つのことです。

**1つ目は、「AIでここまでできる」という事実。**

コミットの半分はAIが書きました。23分間で6つの改善。1コミットで3,390行。5時間でApp Store公開。これらはすべて、git履歴に残った事実です。

**2つ目は、「でも人間の判断はまだ必要」という誠実さ。**

アーキテクチャの選択、App Store審査への対応、UXの微調整——これらは今のところ、AIだけでは完結しません。「AIに80%を任せて、人間が残り20%の判断を下す」。これが現時点での最適解だと、私は実感しています。

もしあなたがプログラミングの経験者で、「AIコーディングを試してみたいけど、実際の開発フローがイメージできない」と感じているなら、ぜひ試してみてください。最初の1コミットで、その威力がわかるはずです。

もしあなたがプログラミング未経験で、「AIでアプリが作れるって本当？」と疑っているなら——半分は本当です。AIがコードを書いてくれるのは事実ですが、「何を作りたいか」「何が良いUXか」を判断するのは、やっぱり人間の仕事です。

「AIに仕事を奪われる」のではなく、「AIと一緒に、一人では作れなかったものを作る」。

それが、コミットの半分をAIと共有した私のリアルな実感です。
