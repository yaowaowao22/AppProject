## 競合調査 — Pushover・ntfy・Pushbulletを使って感じた3つの不満点

### 不満1: セットアップが面倒

Pushoverは有料（$5の買い切り）で、Application APIキーとUser Keyの2つが必要。ntfyはトピック名さえあれば使えますが、認証なしだと他人にトピック名を推測されるリスクがあります。Pushbulletはアカウント登録→OAuth→アクセストークン取得という手順を踏む必要があります。

どのサービスも、「curl一発で通知を送る」までのステップが多い。

### 不満2: 日本語UIがない

エンジニアとしては英語UIで困ることはありません。でも、「これ便利だよ」と非エンジニアの友人に勧めたとき、セットアップの段階で挫折されました。

### 不満3: 通知の見せ方がシンプルすぎる

通知を受け取ったら、それで終わり。通知の履歴を一覧で見たい、優先度で色分けしたい、URLを添付して通知からそのまま開きたい——こういう「ちょっとした使い勝手」が足りませんでした。

この3つの不満を、全部自分で解決すればいい。

そう思ったのが開発のスタート地点です。

## 「ゼロセットアップ・APIキー1つで即利用」というコンセプトに辿り着くまで

不満を裏返すと、コンセプトが見えてきます。

- **セットアップが面倒** → アプリをインストールした瞬間にAPIキーが発行される
- **日本語UIがない** → 完全日本語対応
- **通知の機能が足りない** → 受信箱UI、優先度、URL添付、カテゴリ分け

特にこだわったのは「ゼロセットアップ」です。

アプリを初めて開いたら、画面にAPIキーがもう表示されている。ユーザー登録もメールアドレスの入力もない。そのAPIキーをcurlに貼り付けて実行すれば、3秒後にスマホが鳴る。

この体験を作ることが、最大の差別化だと考えました。

実際、APIキーはアプリ初回起動時に自動生成されます。

```
curl -X POST https://push-api.example.com/api/send \
  -H "Content-Type: application/json" \
  -d '{"token":"あなたのAPIキー","title":"テスト","body":"Hello!"}'
```

これだけ。アカウント作成ゼロ、設定画面ゼロ、最短10秒で通知が届く。

## 技術選定 — 月額0円で運用できるアーキテクチャ

個人開発で最も怖いのは、**ランニングコスト**です。

「月間1,000ユーザーになったらサーバー代が月¥5,000かかります」では、赤字のリスクを常に背負うことになります。だから私は「月額0円で運用できるアーキテクチャ」を最初に設計しました。

### フロントエンド: Expo（React Native）

1つのコードベースでiOSとAndroidの両方に対応できる。これだけで開発工数が半分になります。Expo SDK 54を使用し、EAS Build/Submitでビルドからストア提出までコマンド一発です。

### バックエンド: Cloudflare Workers + KV

ここが運用コスト0円の核心です。

- **Cloudflare Workers**: 無料枠で**1日10万リクエスト**。個人開発アプリなら余裕でカバーできます
- **Cloudflare KV**: キーバリューストアで、ユーザーデータ・使用量カウント・課金状態をすべて管理。無料枠で十分
- **コールドスタートなし**: AWS LambdaやCloud Functionsと違い、Cloudflare Workersはエッジで動くので初回リクエストでも遅延ゼロ

データベースが不要なので、月額固定費がかかるRDSやPlanetScaleは一切使っていません。

### 課金: RevenueCat

App StoreとGoogle Playの課金処理を統一的に扱えるサービスです。レシート検証（「この人は本当に購入したのか？」を確認する処理）をRevenueCatが代行してくれるので、自前のサーバーでAppleやGoogleと通信する必要がありません。

月間売上$2,500まで無料。個人開発の初期フェーズでは一切コストがかかりません。

### プッシュ通知: Expo Push Service

Expo Push Serviceが、Apple（APNs）とGoogle（FCM）の違いを吸収してくれます。サーバーからは統一されたAPIを叩くだけ。しかもこれも無料。

**結果として、月額の固定費はゼロ。** ドメインもCloudflare Workersのサブドメイン（`*.workers.dev`）を使えば無料です。

## フリーミアム設計の全思考プロセス — なぜ「月10通無料 + ¥300買い切り」にしたのか

課金モデルを決めるのは、コードを書くより何倍も悩みました。

### なぜ「買い切り」なのか？ — サブスクを選ばなかった理由

noteで個人開発の収益化について書かれた記事を読むと、そのほとんどが**月額サブスク**を勧めています。「LTV（生涯顧客価値）が高い」「MRR（月次経常収益）で安定する」と。

理論的にはその通りです。でも、私は買い切りを選びました。理由は3つ。

**理由1: ユーティリティアプリはサブスクの心理的ハードルが高い**

「かんたんプッシュ」は、使う人にとっては毎日使うツールですが、使う頻度には波があります。サーバーの監視をセットアップしたら、通知が来るのを待つだけ。能動的に「使っている」感覚がないアプリに月額を払い続けるのは、ユーザーにとってストレスです。

Pushoverが$5の買い切りで成功しているのを見て、確信しました。**ユーティリティ系は買い切りが正解**です。

**理由2: ¥300は「考えずに買える」価格帯**

¥300は缶コーヒー2本分です。

App Storeで¥300のアプリを見たとき、「ちょっと試してみるか」と思えるギリギリの価格。これが¥980だと「元が取れるかな？」と計算が始まります。¥300なら、計算する前に指が「購入」を押している。

心理学でいう「認知的負荷の閾値」を下回る価格設定です。

**理由3: Appleの手数料を考慮した実質的な手取り**

App Storeの手数料は、年間売上$100万以下の小規模開発者なら**15%**（App Store Small Business Program）。

¥300の15% = ¥45が手数料。手取りは**¥255**。

これが¥100だと手取り¥85。単価が低すぎて、課金率が高くても収益が伸びません。¥500だと手取り¥425で美味しいですが、前述の心理的ハードルが上がります。

¥300は「ユーザーが気軽に買えて、開発者にも十分な手取りが残る」バランスポイントでした。

### なぜ「月10通」なのか？ — FREE_LIMIT=10の数値的根拠

無料枠の設定は、多すぎても少なすぎてもダメです。

```ts
// src/types.ts
export const FREE_LIMIT = 10;
```

この「10」という数字には明確な根拠があります。

**根拠1: 「試用」と「実用」の境界線**

プッシュ通知APIを試す場合、セットアップのテスト送信で2〜3通、本番スクリプトの動作確認で2〜3通、合計5〜6通あれば「試す」には十分です。月10通なら、お試しには余裕がある。

一方、実際に運用するとなると、サーバー監視のアラートだけで月10通はすぐに超えます。バッチの完了通知を入れたら到底足りない。

つまり月10通は、**「試すには十分だが、実用するには足りない」絶妙なライン**です。

**根拠2: 競合サービスとの比較**

- Pushover: 10,000通/月（ただし$5の買い切り前提）
- ntfy: 無制限（ただしレート制限あり）
- Pushbullet: 500通/月（無料プラン）

月10通は競合と比べて明らかに少ない。でも、「かんたんプッシュ」の無料枠の目的は「永久に無料で使ってもらうこと」ではなく、**「体験して価値を感じてもらい、¥300で買ってもらうこと」**です。

無料ユーザーが快適すぎると、誰も課金しません。

**根拠3: サーバーコストの防衛線**

Cloudflare Workersの無料枠は1日10万リクエスト。月10通×ユーザー数であれば、かなりのユーザー数でも無料枠に収まります。万が一バズって大量のユーザーが流入しても、サーバーコストが急増しない安心感。

### 二重チェックの設計 — クライアントとサーバーの両方で制限する

使用量チェックは、**クライアント側（アプリ）とサーバー側の両方**で行っています。

クライアント側のチェック（SettingsScreen.tsx）:

```ts
// 送信前にアプリ側でチェック — 即座にユーザーにフィードバック
if (!isPremium && currentUsage.count >= FREE_LIMIT) {
  Alert.alert(
    '送信制限',
    `無料プランの月${FREE_LIMIT}通に達しました。\n設定タブからプレミアムにアップグレードしてください。`
  );
  return;
}
```

なぜ二重にするのか？

クライアント側だけだと、APIを直接叩けば制限を回避できてしまいます（curlで叩けるAPIなので当然です）。サーバー側だけだと、ユーザーは通知を送ろうとしてからエラーを受け取ることになり、体験が悪い。

- **クライアント側**: ユーザー体験のための即座なフィードバック
- **サーバー側**: セキュリティのための最後の砦

```
サーバー側の制限ロジック:
- 無料ユーザー: 月間使用量 >= 10 → HTTP 429エラー
- プレミアムユーザー: 日間使用量 >= 2,000 → HTTP 429エラー
- 使用量はCloudflare KVで管理（月次データは60日TTL、日次データは2日TTLで自動削除）
```

プレミアムユーザーでも**日2,000通の上限**を設けています。これは悪用防止の安全弁です。正当な使い方で日2,000通を超えることはまずありませんが、万が一APIキーが漏洩してスパム送信に使われた場合のダメージを限定するためです。

## RevenueCat課金実装 — たった63行のコード全公開

ここが多くの個人開発者がつまずくポイントだと思います。

「課金の実装って難しそう」——そう思っている方に朗報です。RevenueCatを使えば、課金関連のコードは**全部で63行**です。ファイル1つに収まります。

### purchases.ts — 課金ユーティリティの全体像

まず全体構成を説明します。このファイルには4つの関数があります。

| 関数名 | 役割 | 行数 |
|--------|------|------|
| `initPurchases()` | RevenueCat SDKの初期化 | 約10行 |
| `buyPremium()` | 購入フロー | 約20行 |
| `restorePurchases()` | 購入復元 | 約6行 |
| `checkPremium()` | 課金状態の確認 | 約5行 |

たったこれだけ。

### initPurchases() — 初期化

```ts
const REVENUECAT_IOS_KEY = 'appl_frBDjrHKKZhHlPYwQdmeOgXzzFq';
const REVENUECAT_ANDROID_KEY = 'appl_frBDjrHKKZhHlPYwQdmeOgXzzFq';
const ENTITLEMENT_ID = 'premium';

// アプリ起動時に1回だけ呼ぶ。二重初期化をフラグで防止
export async function initPurchases(): Promise<void> {
  if (initialized) return;
  const key = Platform.OS === 'ios' ? REVENUECAT_IOS_KEY : REVENUECAT_ANDROID_KEY;
  Purchases.configure({ apiKey: key });
  initialized = true;
}
```

RevenueCatのダッシュボードでアプリを登録すると、APIキーが発行されます。iOSとAndroidで同じキーを使っているのは、RevenueCat側で統合管理しているためです（プラットフォームごとに分けることも可能です）。

`initialized`フラグで二重初期化を防いでいます。React Nativeでは`useEffect`が複数回走ることがあるので、この防御は必須です。

### buyPremium() — 購入フロー

ここが課金の核心部分です。

```ts
export async function buyPremium(): Promise<{ success: boolean; error?: string }> {
  // RevenueCatの「Offering」から、利用可能な商品パッケージを取得
  const offerings = await Purchases.getOfferings();
  const pkg = offerings.current?.availablePackages[0];

  // 商品が見つからない場合（RevenueCat側の設定ミスなど）
  if (!pkg) return { success: false, error: '購入可能なプランが見つかりません' };

  // Apple/Googleの購入ダイアログを表示し、結果を受け取る
  const { customerInfo } = await Purchases.purchasePackage(pkg);

  // 「premium」というentitlementが有効かどうかで購入成功を判定
  const hasPremium = customerInfo.entitlements.active[ENTITLEMENT_ID] !== undefined;
  return { success: hasPremium, error: hasPremium ? undefined : '購入処理に失敗しました' };
}
```

非エンジニアの方のために補足すると、`Purchases.getOfferings()`は「この商品を売りますよ」という情報をRevenueCatから取得する処理です。`Purchases.purchasePackage(pkg)`を呼ぶと、iPhoneなら「¥300で購入しますか？」というApple標準のダイアログが表示されます。

ポイントは`offerings.current?.availablePackages[0]`という部分。「現在のOffering（販売商品セット）から最初のパッケージを取る」というシンプルな設計です。商品が1つしかないアプリなので、これで十分。

複数プラン（月額/年額/買い切りの選択肢）を用意する場合は、ここを変える必要がありますが、**最初は1商品で始めて、必要になったら増やす**のが個人開発のコツです。

返り値は`{ success: boolean; error?: string }`という統一フォーマットにしています。呼び出し側で`if (result.success)`と書くだけで済むので、エラーハンドリングがシンプルになります。

### restorePurchases() — 購入復元

```ts
// 端末を変えた場合や、アプリを再インストールした場合に
// 過去の購入を復元する。Apple審査で「購入復元」ボタンは必須
export async function restorePurchases(): Promise<boolean> {
  const info = await Purchases.restorePurchases();
  return info.entitlements.active[ENTITLEMENT_ID] !== undefined;
}
```

**わずか6行**です。しかし、この機能がないとApp Store審査で確実にリジェクトされます。Appleのガイドラインでは、アプリ内課金があるアプリには必ず「購入を復元」する手段を提供することが求められています。

### checkPremium() — 課金状態の確認

```ts
// アプリ起動時に毎回呼び、現在のプレミアム状態を確認する
export async function checkPremium(): Promise<boolean> {
  const info = await Purchases.getCustomerInfo();
  return info.entitlements.active[ENTITLEMENT_ID] !== undefined;
}
```

この関数はアプリ起動時に毎回呼ばれます。RevenueCatから最新の顧客情報を取得し、「premium」エンタイトルメント（権利）が有効かどうかを返します。

端末を変えても、RevenueCatのサーバーに購入履歴が残っているので、ログインなしで自動的にプレミアム状態が復元されます。

### サーバーへの課金状態同期

課金状態はRevenueCatが管理していますが、通知の送信制限はサーバー（Cloudflare Workers）側で行います。つまり、**「この人はプレミアムだから無制限に送信できる」ということをサーバーにも教えてあげる必要がある**のです。

```ts
// 購入完了後＆アプリ起動時に、サーバーへプレミアム状態を通知
const syncPremiumToServer = useCallback(
  (token: string) => {
    fetch(`${API_BASE}/api/premium`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ token }),
    }).catch(() => {});
  },
  []
);
```

`.catch(() => {})`で、ネットワークエラーを握り潰しています。「え、エラー無視していいの？」と思うかもしれませんが、これは意図的です。

サーバーへの同期はベストエフォートで十分です。今回失敗しても、次にアプリを起動したときにまた同期されます。同期に失敗したからといってユーザーの購入が無効になるわけではありません（購入状態はRevenueCatが管理しているので）。

### アプリ起動時の初期化フロー

実際の起動時処理はこうなっています。

```ts
// App.tsx — RevenueCat初期化 + プレミアム状態の同期
useEffect(() => {
  if (apiKeyLoading || !apiKey) return;
  (async () => {
    // 1. RevenueCat SDKを初期化
    await initPurchases();
    // 2. 現在のプレミアム状態を確認
    const hasPremium = await checkPremium();
    // 3. アプリの状態に反映
    setIsPremium(hasPremium);
    // 4. プレミアムならサーバーにも通知
    if (hasPremium) {
      fetch(`${API_BASE}/api/premium`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ token: apiKey }),
      }).catch(() => {});
    }
  })();
}, [apiKey, apiKeyLoading]);
```

この流れで、**ユーザーがアプリを開くたびに、RevenueCatとサーバーの両方の課金状態が最新になります。** 端末を変えてもアプリを再インストールしても、購入が正しく反映されます。

### 課金状態の全画面配信 — React Context

```ts
// UsageContext.tsx — 全画面で isPremium と usage にアクセス可能
interface UsageContextValue {
  usage: UsageInfo;       // { monthKey: "2026-03", count: 7 }
  isPremium: boolean;     // 課金済みかどうか
}

const UsageContext = createContext<UsageContextValue>({
  usage: { monthKey: getCurrentMonthKey(), count: 0 },
  isPremium: false,
});

export const UsageProvider = UsageContext.Provider;
export function useUsage(): UsageContextValue {
  return useContext(UsageContext);
}
```

ReactのContext APIを使って、`isPremium`と`usage`（使用量）をアプリ全体に配信しています。どの画面からでも`useUsage()`を呼ぶだけでプレミアム状態がわかるので、画面ごとにサーバーに問い合わせる必要がありません。

たとえば受信箱画面のヘッダーには、こんなバッジを表示しています。

```tsx
// 受信箱画面のヘッダー — 課金状態で表示を切り替え
<Badge
  label={isPremium ? 'プレミアム' : `無料 ${currentCount}/${FREE_LIMIT}`}
  variant={isPremium ? 'success' : currentCount >= FREE_LIMIT ? 'error' : 'info'}
/>
```

無料ユーザーには「無料 7/10」のように残り通数を表示。上限に達すると赤いバッジに変わります。この**「赤に変わる瞬間」が課金への自然な導線**になっています。わざとらしく「課金してください！」とポップアップを出すのではなく、バッジの色が変わるだけ。でも、赤いバッジは十分に「そろそろアップグレードしないと」と思わせます。

## App Store審査でリジェクトされた話と具体的な対処法

Day 6、期待に胸を膨らませてApp Store審査に提出しました。

結果は**リジェクト**。

正直、「プッシュ通知を送るだけのシンプルなアプリだし、一発で通るだろう」と楽観していました。甘かったです。

### リジェクト理由1: 暗号化に関する申告

App Store Connectにバイナリをアップロードすると、「このアプリは暗号化技術を使用していますか？」と聞かれます。

プッシュ通知APIはHTTPSを使っているので暗号化を「使っている」と言えなくもないのですが、HTTPS通信のみの場合はAppleの暗号化輸出規制の対象外です。

解決策は、`app.json`に1行追加するだけでした。

```json
{
  "expo": {
    "ios": {
      "infoPlist": {
        "ITSAppUsesNonExemptEncryption": false
      }
    }
  }
}
```

`ITSAppUsesNonExemptEncryption`を`false`にすることで、「このアプリは輸出規制に該当する暗号化を使っていません」と宣言できます。これにより、審査提出時の手動での暗号化質問がスキップされます。

**この1行を知っているかどうかで、審査のストレスが大きく変わります。** 個人開発者にはぜひ覚えておいてほしい設定です。

### リジェクト理由2: 購入復元ボタンの不備

前述の通り、Appleは「購入を復元」する手段をアプリ内に提供することを義務づけています。

私は設定画面に復元ボタンを設置しました。

```tsx
// 設定画面の「購入を復元」ボタン
<ListItem
  title="購入を復元"
  subtitle={restoring ? '復元中...' : '別端末での購入を復元'}
  onPress={restoring ? undefined : handleRestorePurchase}
  rightIcon={
    restoring
      ? <ActivityIndicator size="small" color={colors.primary} />
      : <Ionicons name="chevron-forward" size={18} color={colors.textMuted} />
  }
/>
```

### その他の審査対策

| 対策 | 具体的な実装 |
|------|-------------|
| プライバシーポリシー | Cloudflare Workersの`/privacy`エンドポイントに設置 |
| EULA（利用規約） | Appleの標準EULAを利用（自前で作成不要） |
| ビルド番号の管理 | `eas.json`の`autoIncrement: true`で自動インクリメント |
| 3環境のビルド構成 | development（シミュレータ）/ preview（テスター配布）/ production（ストア提出） |

結局、**ビルド番号6**（6回目の提出）でようやく審査を通過しました。最初の5回で不備を潰していった形です。

```json
// eas.json — ビルド構成
{
  "build": {
    "development": { "developmentClient": true, "ios": { "simulator": true } },
    "preview":     { "distribution": "internal", "channel": "preview" },
    "production":  { "autoIncrement": true, "channel": "production" }
  },
  "submit": {
    "production": { "ios": { "ascAppId": "6759830379" } }
  }
}
```

EAS Submitを使えば、ターミナルから`eas submit -p ios`と打つだけでApp Store Connectにバイナリがアップロードされます。Xcode不要。これもExpoの大きなメリットです。

## 個人開発で¥300アプリを売るために私がやった5つのこと

技術的な実装だけでなく、「売るための行動」も重要です。

### 1. App Store Optimization（ASO）を意識したタイトルとサブタイトル

アプリ名は**「かんたんプッシュ」**。

「プッシュ通知」「かんたん」という、ユーザーが検索しそうなキーワードを含めています。英語だと「EasyPush」ですが、日本語ストアでは日本語名の方が検索に引っかかりやすい。

### 2. Qiitaで技術記事を公開 → アプリへの導線

リリース直後にQiitaで「Cloudflare Workersで自作プッシュ通知APIを作った」という技術記事を公開しました。記事の最後にアプリのリンクを貼る形です。

Week 2のDL数が58に跳ねたのは、この記事がQiitaのトレンドに入ったおかげです。**技術記事は最強のマーケティング**だと実感しました。

### 3. Xで開発過程をリアルタイム発信

開発中から「今日はDay 3、プッシュ通知の送受信テストに成功した」のようにポストしていました。リリース当日には「ついにApp Store公開されました！」のポストに30以上のいいねがつきました。

フォロワーが少なくても、**ハッシュタグ（#個人開発 #indiedev）**をつけることで同じ興味を持つ人に届きます。

### 4. 無料枠で「使わせてから売る」

前述の通り、月10通の無料枠が課金への自然な導線になっています。

ポイントは、**無料枠の段階で「有料版と同じ体験」を提供すること**。無料だから機能が制限される、画面にウォーターマークが出る——そういう「無料だと不快」な設計はしていません。唯一の違いは送信通数だけ。

だから、無料で使った人は「もっと使いたい」と思って課金する。「不快を解消するための課金」ではなく「満足を延長するための課金」です。この差は大きい。

### 5. 価格の心理的な見せ方

設定画面の課金ボタンには、こう表示しています。

```
プレミアムにアップグレード — ¥300
```

「¥300/永久」「¥300（買い切り）」ではなく、**シンプルに¥300だけ**。サブスクではないことは、その上の「送信数無制限・買い切り」というテキストで伝えています。

価格と「買い切り」を別の場所に書くことで、まず¥300という安さに目が行き、その後に「しかも買い切りなの？」という追加の安心感が得られる設計です。

## まとめ — 小さく作って小さく売る個人開発の勝ちパターン

この記事で伝えたかったことをまとめます。

**1. アイデアは「自分の不満」から始める**
「かんたんプッシュ」は、私がサーバー監視で感じた不満から生まれました。自分がユーザーなら、何が欲しいかが正確にわかります。

**2. 運用コスト0円のアーキテクチャを選ぶ**
Cloudflare Workersの無料枠、RevenueCatの無料枠、Expo Push Serviceの無料。個人開発では「売れなくても損しない」設計が精神的安定の鍵です。

**3. サブスクより買い切り（ユーティリティ系の場合）**
¥300の買い切りは、ユーザーの心理的ハードルが低く、課金率が高い。個人開発の小さなアプリには最適な課金モデルです。

**4. フリーミアムの無料枠は「試用に十分、実用に不足」に設定する**
月10通は、テストには十分だが運用には足りない。この絶妙なバランスが7.5%の課金率を生みました。

**5. 課金の実装は63行で済む**
RevenueCatを使えば、購入・復元・状態確認がファイル1つ。恐れるべきはコードの複雑さではなく、「始めないこと」です。

**6. 技術記事はマーケティング**
Qiitaの1本の記事がDL数を1.4倍にしました。エンジニアのマーケティングは、ブログを書くことです。

初月の売上¥2,805は、決して大きな金額ではありません。でも、**自分が作ったものにお金を払ってくれる人が11人いた**。この事実は、金額以上の価値がありました。

**今日できる最初の一歩:** ExpoでReact Nativeプロジェクトを作成し、RevenueCatのダッシュボードでアカウントを作る。所要時間15分。コードを書く前に、この2つだけやってみてください。課金の仕組みが思ったよりシンプルだと気づくはずです。
