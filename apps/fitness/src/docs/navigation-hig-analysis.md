# TANREN ナビゲーション HIG 分析レポート

**分析日**: 2026-03-31
**対象**: React Native (Expo) 筋トレ記録アプリ「TANREN」
**基準**: Apple Human Interface Guidelines (HIG) 2025
**制約**: サイドバー（Drawer）は維持する前提

---

## 1. Apple HIG ナビゲーション原則の要約

### 1.1 Tab Bar（タブバー）

HIG はタブバーを **iPhone アプリのプライマリナビゲーション**として最優先に推奨している。

- 画面下部に常時表示され、3〜5 個のトップレベル目的地を提供
- **常時可視** → 発見性（discoverability）が最も高い
- ユーザーは現在地を常に把握でき、1タップで任意のセクションに移動可能
- アイコン + ラベルの併記が必須（アイコンのみは禁止）
- 実績: Airbnb がハンバーガーメニューからタブバーへ移行し、タスク完了速度 **40% 向上**

### 1.2 Sidebar（サイドバー）

HIG における Sidebar は **iPadOS / macOS 向け**のパターンであり、iPhone では異なる扱いを受ける。

- iPadOS: `UISplitViewController` の Primary Column として常時表示可能。コンテンツリストとの2カラム表示が標準
- macOS: ウィンドウ左端に常駐する Source List パターン
- **iPhone**: Sidebar は表示領域が狭く非推奨。Adaptive Layout では iPad の Sidebar が iPhone では Tab Bar に変換される
- HIG は iPhone 向けに Sidebar/Drawer を明示的に推奨していない

### 1.3 Drawer（ハンバーガーメニュー）

Apple は HIG でハンバーガーメニュー（Drawer）を **一度も公式に推奨したことがない**。

- Android の Material Design では Navigation Drawer が公式パターンだが、iOS では非標準
- 主な問題点:
  - **低い発見性**: メニュー項目がハンバーガーアイコンの背後に隠れ、ユーザーが機能を発見できない
  - **追加タップコスト**: すべてのナビゲーションに「開く → 選択」の2ステップが必要
  - **現在地の不明確さ**: Drawer を閉じると現在どのセクションにいるか視覚的手がかりが失われる
  - **片手操作の困難さ**: 左上のハンバーガーアイコンは片手持ち時に最も届きにくい位置
- 歴史的経緯: 2013-2015年頃、Facebook・Spotify・YouTube 等が Drawer → Tab Bar へ移行し、エンゲージメントが大幅改善

### 1.4 Segmented Control（セグメンテッドコントロール）

画面内で **同一データの異なるビュー**を切り替える場合、HIG は Segmented Control を推奨する。

- ナビゲーションではなく「フィルター / ビュー切替」として機能
- 2〜5 個のセグメントが適切
- グローバルナビゲーション（Tab Bar / Sidebar）とは異なるレイヤーの UI
- Tab Bar とは共存可能（Tab Bar = 画面間移動、Segmented Control = 画面内ビュー切替）

## 2. 現状の問題点分析

### 2.1 現在のナビゲーション構造

```
Drawer (ハンバーガーメニュー) — ルートナビゲーション
├── ホーム (HomeScreen)
├── トレーニング (WorkoutStack)
│   ├── ExerciseSelect
│   ├── OrderConfirm
│   ├── ActiveWorkout
│   └── WorkoutComplete (modal)
├── 履歴 (HistoryStack)
│   ├── HistoryScreen ← 画面内タブ: 日別 / 部位別 / 種目別
│   ├── DayDetail
│   └── SessionEdit
├── 月別レポート (MonthlyReportScreen)
├── RM計算機 (RMCalculatorScreen)
├── テンプレート (TemplateManageScreen)
└── 設定 (SettingsScreen)
```

Drawer 項目数: **6 画面 + 設定 = 7 項目**

### 2.2 HIG 観点での問題

#### 問題 1: プライマリナビゲーションが Drawer（非推奨パターン）

TANREN の全画面が Drawer 内に格納されており、iOS ユーザーが期待する Tab Bar が存在しない。これは HIG の最も基本的なナビゲーション原則に反する。

**影響**:
- 新規ユーザーが「履歴」「レポート」等の機能を発見できない可能性
- すべての画面遷移にハンバーガーメニューの開閉が必要（2タップ）
- 片手操作時、左上のハンバーガーアイコンに指が届きにくい

#### 問題 2: Drawer と画面内タブの認知的不整合

HistoryScreen には日別 / 部位別 / 種目別の画面内タブがあるが、グローバルナビゲーションは Drawer（隠れている）。

**認知負荷の分析**:
- ユーザーは「見えるタブ」と「隠れたメニュー」の2つのナビゲーション概念を同時に理解する必要がある
- 「タブがある画面」と「タブがない画面」が混在し、操作の一貫性が損なわれる
- ただし、**画面内タブ自体は Segmented Control として HIG に合致**しており、問題は Drawer との組み合わせにある

#### 問題 3: 7 項目の Drawer は情報過多

Drawer に 7 項目（設定含む）が並ぶことで、使用頻度の高い画面（ホーム、トレーニング、履歴）と低い画面（RM計算機）が同列に並び、優先度の視覚的区別がない。

### 2.3 画面内タブ（HistoryScreen）の評価

HistoryScreen の日別 / 部位別 / 種目別タブは **HIG 的に問題ない**。

- 同一データセット（トレーニング履歴）を異なる切り口で表示する Segmented Control パターン
- 3 セグメントは適切な数（HIG 推奨: 2〜5）
- グローバルナビゲーションとは独立したレイヤーで機能している
- **問題は画面内タブの存在ではなく、グローバルナビゲーションが Drawer であること**

## 3. 推奨ナビゲーションパターン（サイドバー維持前提）

### パターン A: Bottom Tab Bar（主） + Sidebar（副: ダッシュボード）【最推奨】

```
BottomTabBar (主ナビゲーション)
├── ホーム           (barbell)
├── トレーニング     (fitness)
├── 履歴             (time)
├── レポート         (stats-chart)
└── その他           (ellipsis.horizontal.circle)

Sidebar (副: ハンバーガーまたはスワイプで表示)
├── ミニダッシュボード (週間統計)
├── テンプレート管理
├── RM計算機
└── 設定
```

**構造詳細**:
- Tab Bar に使用頻度の高い 4〜5 画面を配置
- Sidebar はダッシュボード表示 + セカンダリ機能のハブとして維持
- Sidebar のハンバーガーアイコンは ScreenHeader 左上に残す
- 「その他」タブは MonthlyReport / RMCalculator / TemplateManage / Settings をリスト表示（iOS 標準の "More" パターン）
- HistoryScreen の画面内タブはそのまま維持

| 観点 | 評価 |
|---|---|
| HIG 準拠度 | **最高** — Tab Bar がプライマリ、Sidebar が補助で理想的 |
| 発見性 | **最高** — 主要機能が常時可視 |
| 認知負荷 | **低** — Tab Bar で現在地が明確、Sidebar は補助情報 |
| 操作一貫性 | **高** — すべての主要画面が 1 タップでアクセス可能 |
| 画面内タブとの整合性 | **高** — Tab Bar（画面間移動）と Segmented Control（画面内切替）で役割が明確に分離 |
| 実装コスト | **高** — `@react-navigation/bottom-tabs` 導入、ナビゲーション構造の大幅変更が必要 |
| Sidebar の活用度 | **中** — ダッシュボード + セカンダリ機能に限定されるが、ミニ統計の表示場所として自然 |

**Pros**:
- iOS ユーザーが最も慣れたパターンで学習コストゼロ
- 主要機能への到達が 1 タップ（現状の 2 タップから改善）
- Sidebar のミニダッシュボードが差別化要素として残る
- Tab Bar の badge（未読数等）で通知を表示可能
- 画面内タブとの役割分担が明確

**Cons**:
- 実装工数が最も大きい（ナビゲーション構造の再設計）
- Drawer と Tab Bar の両方が存在することで、Drawer の存在意義がやや薄れる
- Tab Bar が画面下部の表示領域を常に占有（約 49pt）

---

### パターン B: Sidebar（主, 強化版）+ ホーム画面ハブ化

```
Sidebar (主ナビゲーション — 現状維持 + 強化)
├── ミニダッシュボード (週間統計)
├── ── 主要 ──────────
├── ホーム
├── トレーニング
├── 履歴
├── ── ツール ─────────
├── 月別レポート
├── RM計算機
├── テンプレート
└── 設定

HomeScreen (ハブ画面 — 強化)
├── カレンダー
├── クイックアクション (横スクロール)
│   ├── トレーニング開始
│   ├── 履歴を見る
│   └── 月別レポート
├── 今日のメニュー
└── 最近の記録サマリー
```

**構造詳細**:
- Sidebar の構造は現状維持だが、セクション分け（主要 / ツール）でグルーピングを強化
- HomeScreen をハブ化し、主要機能へのショートカットカードを配置
- ハンバーガーメニューに加え、画面端からのスワイプジェスチャーでも Sidebar を開けるようにする
- HistoryScreen の画面内タブはそのまま維持

| 観点 | 評価 |
|---|---|
| HIG 準拠度 | **低** — Drawer がプライマリのまま（HIG 非推奨） |
| 発見性 | **中** — HomeScreen のショートカットで改善するが、Sidebar 依存は変わらず |
| 認知負荷 | **中** — Sidebar + HomeScreen ショートカット + 画面内タブの 3 層 |
| 操作一貫性 | **中** — HomeScreen 経由と Sidebar 経由の 2 つのナビゲーションパスが存在 |
| 画面内タブとの整合性 | **中** — Drawer（隠）と画面内タブ（可視）の不整合は残る |
| 実装コスト | **低** — 既存構造を維持し、HomeScreen にカードを追加するのみ |
| Sidebar の活用度 | **高** — Sidebar がアプリの中心的 UI として最大限活用される |

**Pros**:
- 実装コストが最も低い（既存ナビゲーション構造を変更しない）
- Sidebar のミニダッシュボードやデザイン（鍛鉄の間の世界観）を最大限活かせる
- HomeScreen のハブ化で Sidebar を開かなくても主要機能にアクセス可能

**Cons**:
- HIG 非準拠のまま（Drawer がプライマリ）
- iOS ユーザーの期待するナビゲーションパターンと乖離
- HomeScreen を経由しないと主要機能にワンタップでアクセスできない
- 「Sidebar と HomeScreen ショートカットのどちらを使えばいいか」というユーザーの迷いが発生

### パターン C: Bottom Tab Bar（ミニ 3 タブ）+ Sidebar（フル機能）

```
BottomTabBar (ミニ — 3タブのみ)
├── ホーム           (barbell)
├── トレーニング     (fitness)
└── 履歴             (time)

Sidebar (フル機能 — ハンバーガーまたはスワイプで表示)
├── ミニダッシュボード
├── ホーム ✓
├── トレーニング ✓
├── 履歴 ✓
├── 月別レポート
├── RM計算機
├── テンプレート
└── 設定
```

**構造詳細**:
- Tab Bar を最小限の 3 タブに絞り、最頻使用画面のみを配置
- Sidebar は全画面へのアクセスを提供するフルナビゲーションとして維持
- Tab Bar と Sidebar で重複する項目（ホーム、トレーニング、履歴）は意図的に許容
- Sidebar の重複項目にはチェックマーク等で「Tab Bar にもある」ことを示す

| 観点 | 評価 |
|---|---|
| HIG 準拠度 | **高** — Tab Bar があるため最低限の HIG 準拠 |
| 発見性 | **高** — コア 3 機能は Tab Bar で常時可視、その他は Sidebar |
| 認知負荷 | **中** — Tab Bar と Sidebar の重複が若干の混乱を招く可能性 |
| 操作一貫性 | **中〜高** — コア機能は Tab Bar、拡張機能は Sidebar で一貫 |
| 画面内タブとの整合性 | **高** — Tab Bar + Segmented Control の組み合わせは標準的 |
| 実装コスト | **中** — Tab Bar の導入は必要だが、Sidebar 構造は維持 |
| Sidebar の活用度 | **高** — Sidebar がセカンダリ + ダッシュボードとして十分に活用される |

**Pros**:
- Tab Bar の導入で HIG 準拠しつつ、Sidebar のフル機能を維持
- Tab Bar を 3 タブに抑えることで画面下部の圧迫感が少ない
- Sidebar が「全機能へのアクセスポイント」として明確な役割を持つ
- 段階的な実装が可能（まず 3 タブ → 将来的に 4-5 タブに拡張）

**Cons**:
- Tab Bar と Sidebar の項目重複がユーザーを混乱させる可能性
- 月別レポート・テンプレート等は依然として Sidebar 経由でしかアクセスできない
- 「なぜ一部の画面だけ Tab Bar にあるのか」という疑問が生じうる

### パターン D: Sidebar + コンテキストバー（画面下部フローティング）

```
Sidebar (主ナビゲーション — 現状維持)
├── ミニダッシュボード
├── 全画面リスト（現状通り）
└── 設定

コンテキストバー (画面下部, 画面に応じて変化)
├── HomeScreen:     [トレーニング開始] [履歴] [レポート]
├── HistoryScreen:  [日別|部位別|種目別] (既存タブ)
├── ActiveWorkout:  [セット記録] [完了]
└── その他の画面:    表示なし
```

**構造詳細**:
- Sidebar をプライマリナビゲーションとして維持
- Tab Bar の代わりに、画面ごとに内容が変わるコンテキストバーを画面下部に配置
- HomeScreen ではクイックアクションボタン、HistoryScreen では既存の画面内タブ、ActiveWorkout ではアクションボタンを表示
- 表示する必要がない画面ではバーを非表示

| 観点 | 評価 |
|---|---|
| HIG 準拠度 | **低** — Tab Bar ではないため HIG のタブバー要件を満たさない |
| 発見性 | **中** — コンテキストに応じたアクションは見えるが、グローバルナビは隠れたまま |
| 認知負荷 | **中〜高** — 画面ごとにバーの内容が変わるため予測しにくい |
| 操作一貫性 | **低** — 画面ごとに異なる操作体系 |
| 画面内タブとの整合性 | **中** — HistoryScreen のタブはコンテキストバーに自然に統合可能 |
| 実装コスト | **中〜高** — コンテキストバーコンポーネントの設計・各画面ごとの表示ロジック |
| Sidebar の活用度 | **高** — Sidebar が唯一のグローバルナビゲーションとして最大活用 |

**Pros**:
- Sidebar のデザイン・世界観を最大限に維持
- 画面ごとに最適なアクションを提示でき、UI がコンテキストに応じて変化
- HistoryScreen の既存タブをコンテキストバーに自然に統合可能

**Cons**:
- HIG 非準拠（Tab Bar ではない）
- 画面ごとに異なるバー内容は学習コストが高い
- カスタム UI のため、React Navigation の標準コンポーネントでは実現しにくい
- 「このアプリのナビゲーションはどう動くのか」をユーザーが予測できない

## 4. 結論と推奨

### 最終推奨: パターン A（Bottom Tab Bar + Sidebar ダッシュボード）

**理由**:

1. **HIG 準拠度が最高**: iOS ユーザーの期待するナビゲーション体験を提供
2. **発見性の劇的改善**: 主要機能が常時可視になり、新規ユーザーの機能発見率が向上
3. **認知負荷の解消**: Tab Bar（画面間移動）と Segmented Control（画面内切替）の役割分離が明確
4. **Sidebar の価値を維持**: ミニダッシュボード（週間統計）は Sidebar ならではの価値であり、補助的な位置づけとして活用可能
5. **業界実績**: Airbnb, Facebook, YouTube 等の大手アプリが同じ移行パスで成功

### 推奨 Tab Bar 構成

```
Tab 1: ホーム        (barbell / barbell-outline)
Tab 2: トレーニング  (fitness / fitness-outline)
Tab 3: 履歴          (time / time-outline)
Tab 4: レポート      (stats-chart / stats-chart-outline)
Tab 5: その他        (ellipsis-horizontal-circle / ellipsis-horizontal-circle)
        ├── 月別レポート
        ├── RM計算機
        ├── テンプレート管理
        └── 設定
```

または、5 タブ目を設定にし、月別レポート・RM計算機・テンプレートは Sidebar に残す:

```
Tab 1: ホーム
Tab 2: トレーニング
Tab 3: 履歴
Tab 4: レポート
Tab 5: 設定

Sidebar (補助)
├── ミニダッシュボード
├── 月別レポート
├── RM計算機
└── テンプレート管理
```

### 段階的移行のすすめ

実装コストが高いため、以下の段階的アプローチを推奨:

1. **Phase 1**: `@react-navigation/bottom-tabs` を導入し、主要 4-5 画面を Tab Bar に配置。Sidebar は維持
2. **Phase 2**: Sidebar の役割をダッシュボード + セカンダリ機能に再定義。Sidebar のミニ統計表示を強化
3. **Phase 3**: ユーザーフィードバックを元に Tab Bar の構成を微調整。必要に応じて Sidebar の項目を増減

### 画面内タブ（HistoryScreen）について

HistoryScreen の日別 / 部位別 / 種目別タブは **変更不要**。Segmented Control パターンとして HIG に合致しており、Tab Bar 導入後も自然に共存する。Tab Bar が「どの画面にいるか」を示し、画面内タブが「同一データのどのビューか」を示すという明確な役割分担が成立する。
