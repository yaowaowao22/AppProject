## 11. 俳句帳 — 詠む・記録する・季語を学ぶ

**総合ランク: B**

### 1. Expo/React Native での実装可能性

| 機能 | 実装手段 | 難易度 |
|------|----------|--------|
| 縦書きプレビュー | `WebView` + CSS `writing-mode: vertical-rl` | 中 |
| 音数カウント (5-7-5) | ひらがな変換ライブラリ (`kuromoji` or `wanakana`) + 文字数カウント | 中 |
| 季語辞典 (2,000語) | バンドル JSON + `expo-sqlite` でインデックス | 低 |
| PDF 縦書き出力 | `react-native-html-to-pdf` + HTML テンプレート (CSS writing-mode) | 中 |
| 句帳管理 | `expo-sqlite` CRUD | 低 |

**最大の技術課題**: React Native には `writing-mode` の直接サポートがない。縦書き表示は WebView (HTML レンダリング) または `react-native-skia` でテキストを90度回転する方法が現実的。`react-native-html-to-pdf` と組み合わせれば PDF も縦書き対応可能。

### 2. 保守費ゼロの実現性

✅ **完全オフライン可能**
- 季語データ: バンドル JSON (2,000語 ≒ 約500KB)
- 日付ベース季語推薦: `new Date()` から月判定 → 決定論的に選出
- 句帳データ: `expo-sqlite` でデバイス保存
- PDF 生成: `react-native-html-to-pdf` はデバイス内処理

### 3. 開発工数見積もり

| フェーズ | 内容 | 期間 |
|----------|------|------|
| 俳句エディタ + 音数カウント | ひらがな変換 + カウンタ UI | 1週間 |
| 季語辞典 (検索・一覧) | JSON → SQLite インデックス + 検索 UI | 1週間 |
| 句帳管理 (タグ・整理) | CRUD + フィルタリング | 1週間 |
| 縦書き PDF 出力 | HTML テンプレート作成 + WebView プレビュー | 1.5週間 |
| UI ポリッシュ (高齢者向け大フォント) | テーマ設定 + アクセシビリティ | 1週間 |
| **合計** | | **約 5.5 週間** |

### 4. Apple 審査通過の難易度

**低リスク** — 教育・創作系アプリで審査問題になる要素なし。コンテンツはすべてバンドルされており課金も買い切り。

### 5. 技術的リスク

- **ひらがな変換精度**: `kuromoji.js` (約10MB) はサイズが大きい。代替として軽量な独自ルールベース変換器を作成するか、`wanakana` で入力をひらがな変換 + モーラ数をカウントする方法が現実的 (アプリサイズを抑えられる)
- **縦書き PDF**: WebView 経由の PDF は改ページ制御が難しい。長い句集では複数ページ対応の実装コストが高い
- **高齢ユーザー**: フォントサイズを Dynamic Type 対応にすること必須

### 6. 推奨技術スタック

```
expo-sqlite              # 句帳 + 季語辞典
wanakana                 # ひらがな変換 (軽量)
react-native-webview     # 縦書きプレビュー
react-native-html-to-pdf # PDF 生成
expo-sharing             # 共有シート
expo-file-system         # ファイル操作
@shopify/flash-list      # 大量リスト描画
```

### 7. 改善提案

- **ひらがな入力モード**: キーボードをひらがな前提に設定し、自動音数カウントの精度を上げる
- **縦書きはプレビューのみ**: 編集は横書き、閲覧・共有時だけ縦書き WebView に切り替えることでコストを抑える
- **句集 PDF はシンプルなレイアウトに限定** (複数ページより1句1画像の方が実装が楽)

## 12. ドット絵工房 — ピクセルアートエディタ

**総合ランク: C**

### 1. Expo/React Native での実装可能性

| 機能 | 実装手段 | 難易度 |
|------|----------|--------|
| ピクセルキャンバス描画 | `react-native-skia` または `expo-gl` | 高 |
| レイヤー機能 (5レイヤー) | Skia レイヤー合成 | 高 |
| アニメーション + GIF 出力 | `gif-encoder` (npm) + FileSystem | 高 |
| PNG/カスタムパレット | Skia Image 操作 | 中 |
| タッチ操作 (精密描画) | `react-native-gesture-handler` PanGesture | 中 |

**最大の技術課題**:
1. **GIF 生成**: React Native では `gif-encoder` 等の JS ライブラリが使えるが、128x128 × 複数フレームのエンコードはメモリ・パフォーマンス問題が発生しやすい
2. **精密タッチ**: 小さなピクセル (8x8〜32x32 グリッド) を指でタップする UX は画面ズーム + ピンチ操作が必須で、Gesture Responder の実装が複雑

### 2. 保守費ゼロの実現性

✅ **オフライン完結可能**
- すべての描画・エクスポートはデバイス内処理
- ただし GIF 生成の JS 実装は大きなファイルで不安定になるリスクあり

### 3. 開発工数見積もり

| フェーズ | 期間 |
|----------|------|
| 基本キャンバス + 単色描画 | 2週間 |
| カラーパレット + ツール (消しゴム・塗りつぶし) | 1週間 |
| レイヤー実装 | 1.5週間 |
| アニメーション + GIF 出力 | 2週間 |
| PNG 書き出し + 壁紙サイズ対応 | 1週間 |
| UI ポリッシュ + パレットプリセット | 1週間 |
| **合計** | **約 8.5 週間** |

### 4. Apple 審査通過の難易度

**低リスク** — クリエイティブツールは審査問題なし。

### 5. 技術的リスク

- **GIF エンコード**: 純 JS 実装は低速・不安定。ネイティブモジュール (`react-native-image-manipulator` 経由 or Custom Native Module) の検討が必要
- **メモリ**: 128x128 × 5レイヤー × アニメーション複数フレームは古い iPhone でメモリ不足になる可能性
- **Skia 学習コスト**: `react-native-skia` は強力だが API の学習コストが高い
- **アプリサイズ**: Skia ライブラリで +10〜15MB 増加

### 6. 推奨技術スタック

```
@shopify/react-native-skia     # 描画エンジン
react-native-gesture-handler   # タッチ・ピンチ操作
expo-file-system               # ファイル保存
expo-media-library             # フォトライブラリ保存
expo-sharing                   # 共有
@react-native-async-storage    # プロジェクト保存
```

GIF 生成は MVP では省略し、**APNG または PNG シーケンスとして出力**に絞ると工数を大幅削減できる。

### 7. 改善提案

- **MVP スコープ**: GIF 出力を v1.0 から外し、PNG + 静止画のみに絞る。アニメーションは v2.0 で追加
- **キャンバスサイズ制限**: v1.0 は最大 64x64 に絞りパフォーマンスリスクを低減
- **グローバル展開適性が高い**: UI テキストを最小化すれば英語・韓国語圏にも展開しやすく、iOS 日本限定より全世界展開でレベニューアップが見込める

| 機能 | 実装手段 | 難易度 |
|------|----------|--------|
| ナンクログリッド UI | `FlatList` グリッド or カスタム `View` 行列 | 中 |
| 漢字入力 (グリッドセル) | `TextInput` + カスタムキーボード | 中 |
| 熟語しりとりパズル | カスタム UI コンポーネント | 低 |
| ヒント機能 | ローカル状態管理 | 低 |
| 日替わりチャレンジ | 日付 seed → 問題インデックス決定 | 低 |

**最大の技術課題**: ナンクロのグリッド UI は「黒マス・白マス」の組み合わせで各セルが独立した `TextInput` を持つ構造になる。500問分の問題データ (グリッドレイアウト + 正解) を JSON で管理するデータ設計が重要。

### 2. 保守費ゼロの実現性

✅ **完全オフライン**
- 問題データはバンドル JSON (500問 × 平均グリッド 15x15 ≒ 約2〜5MB)

⚠️ **コンテンツ追加時はアプリ更新が必要** (問題数が肝なのでアップデート戦略を考慮)

### 3. 開発工数見積もり

| フェーズ | 期間 |
|----------|------|
| 問題データ作成 (JSON設計 + 500問) | 2週間 ※最大工数 |
| ナンクログリッド UI + TextInput | 1.5週間 |
| 熟語しりとりパズル UI | 1週間 |
| 難易度・日替わりシステム | 0.5週間 |
| ヒント + 統計ダッシュボード | 1週間 |
| **合計** | **約 6 週間** |

### 4. Apple 審査通過の難易度

**低リスク** — 教育ゲーム。審査問題なし。脳トレ系は App Store 健康・フィットネスカテゴリに好まれる。

### 5. 技術的リスク

- **問題データ作成が最大ボトルネック**: ナンクロは問題を手動作成するかアルゴリズム生成が必要。500問のクオリティを保つには相当な事前準備コストがかかる
- **漢字入力 UX**: グリッドセルへの1文字入力は iOS の Japanese IME と相性が悪い場合がある。専用の漢字一文字キーボードを実装することを推奨
- **パフォーマンス**: 大きなグリッド (15x15=225セル) の再レンダリング最適化が必要 (`React.memo` 必須)

### 6. 推奨技術スタック

```
expo-sqlite                        # 学習記録
react-native-reanimated            # セルアニメーション
@shopify/flash-list                # 高性能リスト描画
expo-haptics                       # 正解・不正解フィードバック
react-native-gesture-handler       # スワイプ操作
```

### 7. 改善提案

- **問題生成をアルゴリズム化**: ナンクログリッドを自動生成するアルゴリズムを実装すれば問題数を無限に増やせる (工数は+1〜2週間だが長期的価値大)
- **MVP は熟語しりとりパズルのみに絞る**: ナンクロはデータ準備が重いため、最初は熟語しりとりで検証してからナンクロを追加する戦略が有効

## 16. 音リズム — リズムトレーニングゲーム

**総合ランク: B**

### 1. Expo/React Native での実装可能性

| 機能 | 実装手段 | 難易度 |
|------|----------|--------|
| 音声再生 (メトロノーム) | `expo-av` Audio | 中〜高 |
| タップ精度判定 (ms 単位) | `Date.now()` + パターンタイムスタンプ比較 | 中 |
| スコア計算 (Perfect/Great/Good/Miss) | 誤差閾値ベース判定 | 低 |
| BPM 調整 + 段階的速度アップ | タイマー制御 | 低 |
| 上達記録グラフ | `victory-native` | 低 |

**最大の技術課題**: **音声レイテンシー**。React Native の `expo-av` は JS スレッド経由のため音声再生タイミングに 50〜150ms の遅延が生じる。精密なリズムゲームには致命的。

### 2. 保守費ゼロの実現性

✅ **完全オフライン可能**
- メトロノーム音はバンドルアセット (WAV/MP3)
- ゲームロジックはローカル実行
- スコア記録は AsyncStorage

### 3. 開発工数見積もり

| フェーズ | 期間 |
|----------|------|
| 音声レイテンシー検証・対策 | 1週間 |
| リズムパターンデータ設計 (30パターン) | 1週間 |
| タップ判定エンジン | 1週間 |
| UI (パターン表示 + スコア) | 1週間 |
| メトロノーム機能 + BPM 調整 | 0.5週間 |
| 記録・統計 + UI ポリッシュ | 1週間 |
| **合計** | **約 5.5 週間** |

### 4. Apple 審査通過の難易度

**低リスク** — 音楽・教育系アプリ。審査問題なし。

### 5. 技術的リスク

- **音声レイテンシー (最重要リスク)**: `expo-av` の JS スレッドからの音声再生は遅延が不安定。対策として `Audio.Sound.createAsync` のプリロード + `expo-av` の `shouldPlay + positionMillis` を使った事前キューイングが必要。または `react-native-sound` の利用を検討 (Bare Workflow 必要)
- **タップ精度測定**: JS スレッドのブロッキングでタイムスタンプが不正確になるケースがある。`react-native-gesture-handler` の `onBegan` コールバックはネイティブスレッドで呼ばれるため精度が高い
- **バックグラウンド再生**: メトロノームをバックグラウンドで継続再生するには `expo-av` の Audio Session 設定が必要

### 6. 推奨技術スタック

```
expo-av                            # 音声再生 (プリロード必須)
react-native-gesture-handler       # 高精度タップ検出
react-native-reanimated            # ビジュアルフィードバック
expo-haptics                       # タップ触覚フィードバック
victory-native                     # 上達グラフ
@react-native-async-storage        # スコア記録
```

### 7. 改善提案

- **触覚フィードバック (Haptic) をビート通知に活用**: ヘッドフォン使用時でも `expo-haptics` でリズムを体感でき、音声レイテンシーの影響を補完できる
- **MVPではメトロノーム機能に絞る**: タップゲーム部分のレイテンシー問題を解決してからゲームモードを追加するアプローチが安全
- **Bare Workflow への移行を事前検討**: 音声精度への要求が高い場合は最初から Bare Workflow で開発する方が手戻りが少ない

| フェーズ | 期間 |

- **写真のクオリティ保証**: App Review では「機能が期待通りに動作するか」が厳しく見られる。背景除去の精度が低いとリジェクト対象になる可能性

### 5. 技術的リスク

- **グリーンバック購入誘導**: アプリ内でグリーンバック布 (Amazon 等) を紹介することで UX と写真クオリティを両立

- **完全オフラインを諦めて「通信ありでマップ表示、記録データはオフライン」とすれば実装コスト大幅削減**

### 5. 技術的リスク

- **マップパフォーマンス**: 3,000ピンを一度に表示するとパフォーマンス低下。ズームレベルに応じたクラスタリング (`react-native-map-clustering`) が必要

