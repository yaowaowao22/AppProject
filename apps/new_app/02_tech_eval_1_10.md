## アイデア 1: 集中プロ — ポモドーロ × 日本式タイムブロック

### ランク: **A**

### 1. Expo/React Native 実装可能性
- タイマーロジック: `setInterval` + `useRef` のみ。バックグラウンド継続は `expo-background-fetch` + `expo-task-manager` で対応
- 環境音再生: `expo-av` の `Audio.Sound` でバンドルアセットを再生。オフライン完全対応
- グラフ描画: `react-native-chart-kit` または `victory-native` で週次/月次棒グラフ実装
- ローカル通知（ポモドーロ終了アラート）: `expo-notifications` で完結
- **Apple Watch 対応は別途 watchOS アプリが必要**（React Native のみでは不可。Swift/watchOS SDK で別開発）。MVP では省略推奨

### 2. 保守費ゼロの実現性
- **完全オフライン可能**: ○
- 集中スコア・タイムブロックデータ: `expo-sqlite` で永続保存
- 設定値（ポモドーロ時間等）: `@react-native-async-storage/async-storage`
- 環境音ファイル（6種 × 約1〜3MB）はアプリバンドルに同梱。アプリサイズが +10〜20MB 増加する点に注意
- クラウド同期なし → 機種変更時はデータ消失。iCloud バックアップに頼る設計で許容範囲

### 3. 開発工数見積もり
| フェーズ | 内容 | 工数 |
|---------|------|------|
| コア | ポモドーロタイマー + ローカル通知 | 3〜4日 |
| 計画機能 | タイムブロック UI | 4〜5日 |
| グラフ | 日次/週次レポート | 2〜3日 |
| 環境音 | 再生制御 UI | 1〜2日 |
| 仕上げ | UI ブラッシュアップ + EAS ビルド | 3〜4日 |
| **合計** | | **約 3 週間** |

Apple Watch 対応を追加する場合は +2〜3週間。

### 4. Apple 審査通過の難易度
- リスク: **低**
- バックグラウンドオーディオ再生の `UIBackgroundModes` を `app.json` に正しく設定しないと審査リジェクトの可能性あり（設定すれば問題なし）
- ローカル通知のみで外部通信なし → プライバシー審査も容易

### 5. 技術的リスク
- **バックグラウンドタイマー**: iOS はバックグラウンドで `setInterval` が正確に動作しない場合がある。開始時刻を保存して復帰時に経過時間を計算する方式（`Date.now()` ベース）で対処
- 環境音のバンドルサイズ: MP3/AAC 圧縮で管理。App Store の 200MB 制限に注意
- Apple Watch 対応: Expo Managed Workflow では不可。必要なら Bare Workflow へ移行

### 6. 推奨技術スタック
```
expo-av              # 環境音再生
expo-notifications   # ポモドーロ終了通知
expo-sqlite          # 集中ログ永続化
expo-task-manager    # バックグラウンドタスク
react-native-chart-kit  # グラフ描画
@react-native-async-storage/async-storage  # 設定値
```

### 7. 改善提案
- **Apple Watch は MVP から除外**し、v1.0 では iPhone 単体で完結させる
- 環境音は「3種 + あとで追加」の構成にしてバンドルサイズを抑える
- タイムブロック計画機能は複雑になりがち。「今日のフォーカスセッション数」のシンプルな記録から始め、段階的に拡張するのが安全

## アイデア 2: 議事録カード — 会議メモ構造化テンプレート

### ランク: **A**

### 1. Expo/React Native 実装可能性
- 構造化フォーム: React Hook Form + Zod で型安全なフォーム実装
- 全文検索: `expo-sqlite` の FTS5 拡張 または `Fuse.js`（ライブラリ）で十分対応可能
- PDF 出力: `expo-print` (Expo 公式) で HTML→PDF 変換。`react-native-html-to-pdf` も選択肢
- 共有: `expo-sharing` でシステム共有シートを呼び出し

### 2. 保守費ゼロの実現性
- **完全オフライン可能**: ○
- SQLite で議事録データを構造化保存（テーブル設計: meetings, todos, tags）
- PDF は `expo-print` でデバイス内生成 → `expo-file-system` で一時ファイルに保存 → 共有
- 全文検索はローカルインデックス。1,000件以上でも SQLite FTS5 なら高速

### 3. 開発工数見積もり
| フェーズ | 内容 | 工数 |
|---------|------|------|
| DB設計 | SQLite スキーマ + CRUD | 2〜3日 |
| 入力フォーム | 構造化テンプレート UI | 3〜4日 |
| 一覧・検索 | タグ/プロジェクト分類 + 検索 | 3〜4日 |
| TODO管理 | TODO 抽出 + 期限表示 | 2〜3日 |
| エクスポート | PDF/テキスト出力 | 2〜3日 |
| 仕上げ | | 2〜3日 |
| **合計** | | **約 2〜3 週間** |

### 4. Apple 審査通過の難易度
- リスク: **低**
- ファイルアクセスは `expo-file-system` の `cacheDirectory` に限定すれば審査問題なし

### 5. 技術的リスク
- **PDF の日本語フォント**: `expo-print` はデバイスシステムフォントを使用するため日本語は問題なし。カスタムフォントを PDF に埋め込む場合は別途対応が必要
- **検索パフォーマンス**: データ件数が数百件の範囲では Fuse.js で十分。1,000件超なら SQLite FTS5 を使用

### 6. 推奨技術スタック
```

expo-print           # HTML→PDF 変換（Expo 公式、Managed Workflow 対応）

expo-file-system     # 一時ファイル管理
Fuse.js              # ローカル全文検索
react-hook-form      # フォーム管理
```

### 7. 改善提案
- TODO 抽出機能は「チェックしてTODO登録」のシンプル実装から始め、自動抽出（テキスト解析）は v2 で
- PDF テンプレートを複数用意（シンプル版/フォーマル版）するとビジネス訴求が強まる

## アイデア 4: 水リズム — 水分補給リマインダー＆記録

### ランク: **B**

### 1. Expo/React Native 実装可能性
- コア機能（記録+通知+グラフ）: 完全対応可能
- **ウィジェット**: iOS ホーム画面ウィジェットは **Expo Managed Workflow では未対応**。`react-native-widgetkit` を使用するには **Bare Workflow** への移行が必須
- Bare Workflow 移行後も `WidgetKit` 拡張は Swift で別途実装が必要

### 2. 保守費ゼロの実現性
- **コア機能はオフライン完全可能**: ○
- 飲水記録: `AsyncStorage` (JSON配列) で十分
- 推奨量計算: ローカルロジック（体重 × 30〜35ml/kg）
- **ウィジェット**: WidgetKit は App Groups を使って本体アプリとデータ共有。Bare Workflow + EAS Build で対応可能だが設定複雑

### 3. 開発工数見積もり
| フェーズ | 内容 | 工数 |
|---------|------|------|
| コア | 推奨量計算 + 記録 UI | 2〜3日 |
| 通知 | カスタム間隔リマインダー | 1〜2日 |
| グラフ | 日次/週次達成率 | 2〜3日 |
| ウィジェット | Bare移行 + Swift WidgetKit実装 | 5〜7日 |
| 仕上げ | | 2〜3日 |
| **合計（ウィジェットあり）** | | **約 3 週間** |
| **合計（ウィジェットなし）** | | **約 1.5 週間** |

### 4. Apple 審査通過の難易度
- リスク: **低**
- WidgetKit 使用時は App Groups エンタイトルメントが必要。EAS Build で設定可能

### 5. 技術的リスク
- **ウィジェット実装の複雑性**: WidgetKit は Swift で書く必要があり、React Native 開発者には学習コストが高い。`react-native-widgetkit` ライブラリも完全な抽象化はできていない
- ウィジェットを省けばランク A に相当する難易度

### 6. 推奨技術スタック
```
@react-native-async-storage/async-storage  # 飲水記録

react-native-chart-kit   # 達成率グラフ
react-native-widgetkit   # ホーム画面ウィジェット（Bare必須）
```

### 7. 改善提案
- **MVP ではウィジェットを省いてリリース**。ウィジェットは「v1.1 アップデート」としてレビュー後に追加
- ウィジェットの代替として Apple Watch の Complications 対応も同様に v2 で検討
- ウォッチ/ウィジェットなしでも「通知 + アプリ内記録」で十分な差別化が可能

## アイデア 7: 読み方ドリル — 難読漢字＆地名クイズ

### ランク: **B**

### 1. Expo/React Native 実装可能性
- **4択モード**: 完全対応可能。4択 Quiz UI はシンプル
- **手書き入力モード**: ここが技術的ハードル。オフライン手書き文字認識は以下の選択肢:
  - Option A: `@react-native-ml-kit/text-recognition`（Google ML Kit）→ **インターネット接続不要・オフライン対応**だが精度は限定的
  - Option B: Apple Vision Framework（`react-native-vision-camera` + `react-native-vision-camera-face-detector` ではなく独自プラグイン）→ **iOS のみ対応、Bare Workflow 必須**
  - Option C: 手書きキャンバス + ひらがな入力に限定（漢字の手書き認識を省略）

### 2. 保守費ゼロの実現性
- 4択モードは **完全オフライン可能**: ○
- 問題データ (3,000問): JSON バンドル。約 300〜500KB で管理可能
- 手書き認識に外部 API（Google Cloud Vision 等）を使う場合はオフライン不可 → **絶対に使わない**こと
- 日替わりチャレンジ: `Date.now()` ベースのシード値で決定論的生成 ✓

### 3. 開発工数見積もり
| フェーズ | 内容 | 工数 |
|---------|------|------|
| データ | 3,000問 JSON 作成 | 5〜7日 |
| 4択 Quiz | ランク制度 + 復習リスト | 4〜5日 |
| 日替わり | シードベース日替わり10問 | 1〜2日 |
| 手書き | キャンバス + ML Kit 認識 | 5〜7日 |
| 仕上げ | | 2〜3日 |
| **合計（手書きあり）** | | **約 3〜4 週間** |
| **合計（4択のみ）** | | **約 2 週間** |

### 4. Apple 審査通過の難易度
- リスク: **低**
- ML Kit 使用でもオフライン処理なら問題なし

### 5. 技術的リスク
- **手書き認識の精度**: ML Kit の日本語手書き認識は常用漢字では良好だが、難読漢字（非常用）では認識率が下がる可能性。ユーザーフラストレーションの原因になりうる
- **3,000問のデータ品質**: 誤りが多いと低評価レビューに直結。著作権フリーのデータソース選定が重要（国語辞典、文化庁データ等）

### 6. 推奨技術スタック

## アイデア 10: 冠婚葬祭マナー帳 — ご祝儀・香典・服装ガイド

### ランク: **A**
