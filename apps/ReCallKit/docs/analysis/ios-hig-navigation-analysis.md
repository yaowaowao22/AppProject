# ReCallKit — Apple HIG ナビゲーション設計分析

> 作成日: 2026-03-31
> 対象: ReCallKit 現行設計（ios-uiux-spec.md / ddp-sidebar.md / web-design-spec-sidebar.md）
> 準拠基準: Apple Human Interface Guidelines — Navigation / Tab bars / Sidebars

---

## 0. 現行設計の概要

ReCallKit は以下の2つのナビゲーション要素を併用する設計となっている:

| 要素 | 仕様 | 役割 |
|------|------|------|
| **ボトムタブバー（4タブ）** | Today / Library / Map / Journal | トップレベルのセクション切替 |
| **左ドロワーサイドバー** | タグ・コレクション・知識カテゴリの階層表示 | フィルタリング・知識体系のナビゲーション |

サイドバーの開閉手段:
- 左端エッジスワイプ
- ハンバーガーメニューアイコンのタップ
- メインコンテンツ領域タップで閉じる

---

## 1. タブバー指針との適合性

### HIG原則の要約

| HIG原則 | 内容 |
|---------|------|
| **用途** | アプリのトップレベルセクション間のフラットな切替に使用する |
| **タブ数** | 3〜5個が推奨。6個以上は "More" タブが必要となり体験が劣化する |
| **常時表示** | タブバーはすべての画面で常時表示し、ユーザーが現在地を常に把握できるようにする |
| **均等重要性** | 各タブは同等の重要度を持つトップレベルカテゴリであるべき |
| **アイコン+ラベル** | アイコンのみは認知負荷が高い。ラベルを必ず併記する |

### ReCallKit現設計との評価

| 評価項目 | 現設計 | HIG適合 | 評価 |
|---------|--------|---------|------|
| タブ数 | 4タブ（Today / Library / Map / Journal） | 3〜5の推奨範囲内 | **推奨** |
| 常時表示 | 全画面でタブバー表示（Review/QuizModeはフルスクリーンモーダルで隠れるが適切） | フルスクリーンモーダル以外で常時表示は正しい | **推奨** |
| ラベル併記 | SF Symbol + 日本語ラベル併記 | HIGの推奨通り | **推奨** |
| アクティブ状態 | Recall Amber（ソリッド） / 非アクティブ: secondaryLabel（ライン） | 明確な視覚的区別あり | **推奨** |
| タブ間の均等性 | Today（日常起点）/ Library（コア機能）/ Map（Phase 2）/ Journal（Phase 2） | Map・JournalがPhase 2でコンテンツ不足の可能性。ただし設計としては妥当 | **許容範囲** |

**小括:** タブバー設計はHIGにほぼ完全に準拠している。

## 2. サイドバー / ドロワー指針との適合性

### HIG原則の要約

| HIG原則 | 内容 |
|---------|------|
| **iPadOS Sidebar** | iPadOSではSplit Viewの左カラムとしてSidebarが推奨される。常時表示または回転時に自動表示 |
| **iPhoneでの非推奨** | iPhoneではSidebar/ドロワーは推奨されない。画面が狭く、隠れたナビゲーションは発見性（Discoverability）を著しく損なう |
| **ハンバーガーメニュー** | Appleは公式にハンバーガーメニュー（≡）パターンをiPhoneで推奨していない。隠れたナビゲーションはユーザーが機能の存在を認識できないリスクがある |
| **エッジスワイプの競合** | 左端エッジスワイプはiOSの「戻る」ジェスチャーと競合する。NavigationStack内の画面では戻るジェスチャーが優先され、サイドバー開閉が不安定になる |

### AppleがiPhoneドロワーを推奨しない理由

1. **発見性の欠如**: ハンバーガーアイコンの裏に隠れた機能は、ユーザーの50%以上が認識しないという調査結果が複数存在する（A/Bテスト事例: Spotifyのハンバーガー→タブバー移行で利用率向上）
2. **片手操作の困難**: iPhoneの大型化に伴い、左上のハンバーガーアイコンは親指が届きにくい「不到達領域」に位置する
3. **ジェスチャー競合**: iOS標準の「左端スワイプ = 戻る」と「左端スワイプ = サイドバーを開く」が競合し、ユーザーの意図と異なる動作が発生する
4. **認知負荷**: タブバー（常時表示）+ ドロワー（隠れたナビゲーション）の2系統は、ユーザーが「どこに何があるか」を学習するコストを倍増させる

### ReCallKit現設計との評価

| 評価項目 | 現設計 | HIG適合 | 評価 |
|---------|--------|---------|------|
| iPhoneでのドロワー使用 | 左からスライドインするドロワー型オーバーレイ | HIGは非推奨 | **問題あり** |
| ハンバーガーアイコン | サイドバー開閉トリガーとして使用 | Apple純正アプリに前例なし | **問題あり** |
| エッジスワイプ競合 | 左端エッジスワイプでサイドバーを開く | NavigationStack内の「戻る」ジェスチャーと競合 | **問題あり** |
| 隠れたナビゲーション | タグ・コレクション・知識カテゴリがドロワー内に隠蔽 | 発見性の重大な低下 | **問題あり** |
| iPad SplitView展開性 | ddp-sidebar.md で将来的なSplitView常時表示を言及 | 方向性自体はHIG準拠 | **許容範囲** |

**小括:** iPhoneにおける左ドロワーサイドバーの使用はHIGの指針と明確に乖離している。

## 3. Tab Bar + Drawer 共存パターンの是非

### Apple純正アプリのナビゲーション構造

| Apple純正アプリ | ナビゲーション構造 | ドロワー使用 |
|---------------|------------------|:----------:|
| **メール** | タブバー（なし）+ NavigationStack内のMailbox一覧 → メール一覧 → 詳細 | **不使用** |
| **ファイル** | タブバー（最近・ブラウズ・共有）+ NavigationStack内のフォルダ階層 | **不使用** |
| **ミュージック** | タブバー（今すぐ聴く・見つける・ラジオ・ライブラリ・検索）| **不使用** |
| **写真** | タブバー（ライブラリ・For You・アルバム・検索）+ 画面内フィルタ | **不使用** |
| **メモ** | NavigationStack（フォルダ → メモ一覧 → 詳細）。iPadではSplitView | **不使用** |
| **Safari** | タブバー（なし）+ 画面内タブグループ管理 | **不使用** |
| **App Store** | タブバー（Today・ゲーム・App・Arcade・検索）| **不使用** |

**重要な事実:** Apple純正のiPhoneアプリで「タブバー + 左ドロワー」を併用しているものは **1つも存在しない**。

### サードパーティアプリの動向

- **Gmail / Google系**: かつてはハンバーガー+ドロワーを使用していたが、2020年以降タブバーへの移行を進めている
- **Spotify**: 2015年にハンバーガーメニューを廃止しタブバーに移行。利用率が大幅に向上
- **Facebook**: タブバー中心に完全移行済み
- **Slack**: iPadではSplitView、iPhoneではタブバー中心

**結論:** Tab Bar + Drawer の共存はiOSエコシステムにおいて明確な **アンチパターン** であり、HIGの原則に反する。

## 4. 代替パターン候補

ReCallKitのサイドバーが担っている機能（タグ・コレクション・知識カテゴリによるフィルタリング）を、iOS標準パターンで実現する代替案を以下に示す。

### 代替案A: Library画面内フィルタリング（セグメンテッドコントロール + チップ）

サイドバーの機能をLibrary画面内のフィルタUIに統合する。

```
┌──────────────────────────────────────┐
│ Nav: "ライブラリ" (Large Title)       │
│──────────────────────────────────────│
│  🔍 検索...                          │
│──────────────────────────────────────│
│  [すべて | URL | テキスト | スクショ] │  ← セグメンテッドコントロール
│──────────────────────────────────────│
│  タグ:                               │
│  [フロントエンド] [DB] [心理学] [+3]  │  ← 横スクロールチップ
│──────────────────────────────────────│
│  （アイテム一覧）                     │
└──────────────────────────────────────┘
```

| 観点 | 評価 |
|------|------|
| **Pros** | HIG完全準拠。既存ios-uiux-spec.mdのLibrary画面設計と一致。フィルタが常時表示で発見性が高い。iOS標準UIKit/SwiftUIコンポーネントで実装可能 |
| **Cons** | タグが大量（20+）になると横スクロールチップの操作性が低下。コレクション（ユーザー定義フォルダ）の階層表現が困難 |
| **HIG適合** | **推奨** |

### 代替案B: モーダルフィルタシート

フィルタ条件をボトムシート（`.medium` detent）で表示する。Apple純正の「写真」アプリのフィルタに近いパターン。

```
┌──────────────────────────────────────┐
│ Nav: "ライブラリ"          🔍  ≡     │  ← ≡はフィルタアイコン(line.3.horizontal.decrease.circle)
│──────────────────────────────────────│
│  適用中: #フロントエンド, URL         │  ← 適用中フィルタのサマリー表示
│──────────────────────────────────────│
│  （フィルタ適用後のアイテム一覧）      │
│                                      │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│ ┌──────────────────────────────────┐ │ ← ボトムシート(.medium)
│ │ フィルタ              [リセット] │ │
│ │                                  │ │
│ │ 種類:                            │ │
│ │ [すべて] [URL] [テキスト] [スクショ] │
│ │                                  │ │
│ │ タグ:                            │ │
│ │ [フロントエンド✓] [DB] [心理学]  │ │
│ │ [プログラミング] [デザイン] [AWS] │ │
│ │                                  │ │
│ │ コレクション:                    │ │
│ │ [技術メモ] [読書メモ] [仕事]     │ │
│ │                                  │ │
│ │ [適用する]                       │ │
│ └──────────────────────────────────┘ │
└──────────────────────────────────────┘
```

| 観点 | 評価 |
|------|------|
| **Pros** | 大量のタグ・コレクションを整理して表示可能。ボトムシートはiOS 15+の標準パターンで操作が直感的。複合条件フィルタ（タグAND種類）が自然に表現できる。画面遷移なしで適用 |
| **Cons** | フィルタ操作にワンタップ余分にかかる（アイコンタップ→シート表示）。常時表示ではないため、頻繁にフィルタを切り替えるユースケースではやや重い |
| **HIG適合** | **推奨** |

### 代替案C: NavigationStack内の階層ナビゲーション（Appleメモ方式）

Library タブ内で「コレクション一覧 → アイテム一覧」の階層遷移を採用する。Apple純正の「メモ」アプリと同じ構造。

```
画面1: コレクション一覧
┌──────────────────────────────────────┐
│ Nav: "ライブラリ" (Large Title)       │
│──────────────────────────────────────│
│  🔍 検索...                          │
│──────────────────────────────────────│
│  すべてのアイテム              (52) > │
│  ──────────────────────────────────  │
│  最近の保存                    (8)  > │
│  ──────────────────────────────────  │
│  コレクション                        │
│  ├── 技術メモ                 (23) > │
│  ├── 読書メモ                 (15) > │
│  └── 仕事                     (9)  > │
│  ──────────────────────────────────  │
│  タグ                                │
│  ├── #フロントエンド           (12) > │
│  ├── #データベース              (8) > │
│  └── #心理学                    (5) > │
└──────────────────────────────────────┘

画面2: 選択したコレクション/タグ内のアイテム一覧
┌──────────────────────────────────────┐
│ < ライブラリ   "技術メモ"            │
│──────────────────────────────────────│
│  （フィルタ適用済みアイテム一覧）      │
└──────────────────────────────────────┘
```

| 観点 | 評価 |
|------|------|
| **Pros** | Apple純正アプリ（メモ・ファイル）と完全に同じ構造でユーザーの学習コストがゼロ。エッジスワイプの「戻る」ジェスチャーが自然に機能。コレクションとタグの階層を無制限に表現可能 |
| **Cons** | 画面遷移が増えるため、「全アイテムを見ながらタグで絞る」ような横断的フィルタリングがやや煩雑。現在のサイドバーの「一望性」が失われる |
| **HIG適合** | **推奨** |

### 代替案D: 案A + 案Bのハイブリッド（推奨）

頻用フィルタ（種類セグメント + タグチップ横スクロール）は画面内に常時表示し、詳細なフィルタリング（複合条件・コレクション選択）はボトムシートに配置する。

```
┌──────────────────────────────────────┐
│ Nav: "ライブラリ"           🔍  ⫶    │  ← ⫶は詳細フィルタアイコン
│──────────────────────────────────────│
│  [すべて | URL | テキスト | スクショ] │  ← 常時表示セグメント
│  [フロントエンド] [DB] [心理学] [+3]  │  ← 常時表示タグチップ
│──────────────────────────────────────│
│  （アイテム一覧）                     │
│                                      │
│  ⫶タップ → ボトムシートで詳細フィルタ │
│  （コレクション選択、複合条件、         │
│    日付範囲、復習状態など）             │
└──────────────────────────────────────┘
```

| 観点 | 評価 |
|------|------|
| **Pros** | 頻用操作は1タップ、高度な操作はボトムシート。発見性と操作効率を両立。iOS標準パターンの組み合わせで違和感なし。サイドバーが担っていた全機能をカバー可能 |
| **Cons** | UIの情報密度がやや高くなる。タグチップが多い場合の表示領域管理が必要 |
| **HIG適合** | **推奨** |

## 5. iPadOS対応を将来考慮する場合

### HIG iPadOSナビゲーション原則

iPadOS では以下のパターンが推奨される:

1. **UISplitViewController（2カラム / 3カラム）**: プライマリカラム（Sidebar）+ セカンダリカラム（コンテンツ）。iPadOS 14以降のSidebarスタイルが標準
2. **TabView + Sidebar 統合（iPadOS 18+）**: `TabView`が自動的にSidebarに変換される。iPhone ではタブバー、iPad ではサイドバーとして表示される同一コードベース

### ReCallKitへの適用可能性

```
iPhone:
  Tab Bar [Today | Library | Map | Journal]
      └── Library 内にフィルタUI（案A/B/D）

iPad (Compact):
  Tab Bar（iPhone と同じ）

iPad (Regular):
  Split View
  ┌─────────────┬──────────────────────────┐
  │  Sidebar     │  Main Content            │
  │              │                          │
  │  Today       │  （選択された画面）       │
  │  Library     │                          │
  │    すべて    │                          │
  │    コレクション│                         │
  │      技術メモ │                          │
  │      読書メモ │                          │
  │    タグ       │                          │
  │  Map         │                          │
  │  Journal     │                          │
  └─────────────┴──────────────────────────┘
```

**重要:** iPadOS 18+の `TabView` Sidebar統合を使えば、iPhoneでは4タブのタブバー、iPadでは同じ内容がサイドバーに自動展開される。**iPhoneで無理にドロワーサイドバーを実装する必要がない**。

| 評価項目 | 評価 |
|---------|------|
| 将来のiPadOS Split View展開 | **推奨** — ddp-sidebar.md で言及されている方向性はHIG準拠。ただし実現手段はiPhoneドロワーではなく、iPadOS固有のSplit View / TabView Sidebar統合で行うべき |

## 6. 総合評価サマリー

### ナビゲーション要素別評価

| # | ナビゲーション要素 | 評価 | 根拠 |
|---|-------------------|------|------|
| 1 | ボトムタブバー（4タブ） | **推奨** | タブ数・常時表示・ラベル併記すべてHIG準拠 |
| 2 | 左ドロワーサイドバー（iPhone） | **問題あり** | Apple純正に前例なし。発見性・ジェスチャー競合・認知負荷の問題 |
| 3 | ハンバーガーメニューアイコン | **問題あり** | HIGが推奨しないパターン。片手操作困難 |
| 4 | 左端エッジスワイプ（サイドバー開閉） | **問題あり** | iOS標準「戻る」ジェスチャーと競合 |
| 5 | Library画面内フィルタ（セグメント+チップ） | **推奨** | ios-uiux-spec.md に既に設計あり。HIG標準パターン |
| 6 | ボトムシートフィルタ | **推奨** | iOS 15+ 標準。複雑なフィルタ条件に最適 |
| 7 | NavigationStack階層遷移 | **推奨** | Apple メモ/ファイルアプリと同構造 |
| 8 | iPad SplitView / TabView Sidebar | **推奨** | iPadOS標準。将来対応の正しい方向 |

### 乖離ポイントまとめ

| 乖離 | 重大度 | 影響 |
|------|:------:|------|
| iPhoneでの左ドロワー使用 | **高** | 発見性の著しい低下。ユーザーがフィルタ機能の存在に気づかないリスク |
| エッジスワイプ競合 | **高** | NavigationStack内の「戻る」とサイドバー開閉が衝突し、予測不能な動作が発生 |
| ハンバーガーメニュー使用 | **中** | Apple純正アプリに前例なく、iOSユーザーの期待と乖離 |
| Tab + Drawer 二重ナビゲーション | **中** | 認知負荷の増大。「どこに何があるか」の学習コスト倍増 |

### 推奨アクション

1. **iPhoneでの左ドロワーサイドバーを廃止する**
2. サイドバーが担っていたフィルタ機能は **代替案D（Library画面内チップ + ボトムシート）** で吸収する
3. ddp-sidebar.md のデザイン哲学（余白の沈黙・骨格の精度）は、Library画面のフィルタUI設計に継承する
4. iPad対応時は **TabView Sidebar統合** を採用し、iPhoneコードベースから自然に展開する
